75 This tool does testing of a TriBITS-based project using CTest and this script
76 can actually do the push itself using git in a safe way. In fact, it is
77 recommended that one uses this script to push since it will amend the last
78 commit message with a (minimal) summary of the builds and tests run with
79 results and/or send out a summary email about the builds/tests performed.

88

90 
91  # 1.a) See what files are changed, newly added, etc.
92  $ git status
93 
94  # 1.b) Stage the files you want to commit
95  $ git stage <files you want to commit>
96 
97  # 1.c) Create your local commits
98  $ git commit -- SOMETHING
99  $ git commit -- SOMETHING_ELSE
100  ...
101

103  $ git stash

108  NOTE: When multiple repos are involved, use the 'gitdist' command instead of
109  'git'. This script is provided at tribits/python_utils/gitdist. See
110  gitdist --help for details.
111

113 
114  $ cd $PROJECT_HOME
115  $ git local-stat | less # Look at the full status of local repo
116  $ git diff --name-status HEAD ^@{u} # [Optional] Look at the files that have changed

209

211

219

221

225 
226  NOTE: The automatic enable behavior can be overridden or modified using the

231 extra builds specified with --st-extra-builds and --extra-builds):
232

234  the packages that have changed and all of the packages that depend on these

237  --local-do-all is set.)

240  -j or --make-options). (done if --build, --do-all, or --local-do-all is set.)
241 
242  4.c) Run all BASIC tests for enabled packages. (done if --test, --do-all,
243  or --local-do-all is set.)
244 
245  4.d) Analyze the results of the pull, configure, build, and tests and send
246  email about results. (emails only sent out if --send-emails-to!="")
247 
248 5) Do final pull and rebase, append test results to last commit message, and
249 push (done if --push or --do-all is set)
250 
251  5.a) Do a final 'git pull' (done if --pull or --do-all is set)
252 
253  5.b) Do 'git rebase <remoterepo>/<remotebranch>' (done if --rebase is set)
254 
255  5.c) Amend commit message of the most recent commit with the summary of the
256  testing performed. (done if --append-test-results is set.)
257

336 ----------------------------
337 
338 (*) Basic full testing with integrating with global repo(s) without push:
339 
340  ../checkin-test.py --do-all
341 
342  NOTE: This will result in a set of emails getting sent to your email address
343  for the different configurations and an overall push readiness status email.
344 
345  NOTE: If everything passed, you can follow this up with a --push (see
346  below).
347 
348 (*) Basic full testing with integrating with local repo and push:
349 
350  ../checkin-test.py --do-all --push
351 
352  NOTE: By default this will rebase your local commits and amend the last
353  commit with a short summary of test results. This is appropriate for
354  pushing commits that only exist in your local repo and are not shared with
355  any remote repo.
356 
357 (*) Push to global repo after a completed set of tests have finished:
358 
359  ../checkin-test.py [other options] --push
360 
361  NOTE: This will pick up the results for the last completed test runs with
362  [other options] and append the results of those tests to the log of the most
363  recent commit.
364 
365  NOTE: Take the action options for the prior run and replace --do-all with
366  --push but keep all of the rest of the options the same. For example, if
367  you did:
368 
369  ../checkin-test.py --enable-packages=Blah --default-builds=MPI_DEBUG --do-all
370 
371  then follow that up with:
372 
373  ../checkin-test.py --enable-packages=Blah --default-builds=MPI_DEBUG --push
374 
375  NOTE: This is a common use case when some tests are failing which aborted
376  the initial push but you determine it is okay to push anyway and do so with
377  --force-push.
378 
379 (*) Test only the packages modified and not the forward dependent packages:
380 
381  ../checkin-test.py --do-all --no-enable-fwd-packages
382 
383  NOTE: This is a safe thing to do when only tests in the modified packages
384  are changed and not library code. This can speed up the testing process and
385  is to be preferred over not running this script at all. It would be very
386  hard to make this script automatically determine if only test code has
387  changed because every package does not follow a set pattern for
388  tests and test code.
389 
390 (*) Run the most important default (e.g. MPI_DEBUG) build/test only:
391 
392  ../checkin-test.py --do-all --default-builds=MPI_DEBUG
393 
394 (*) The minimum acceptable testing when code has been changed:
395 
396  ../checkin-test.py \
397  --do-all --enable-all-packages=off --no-enable-fwd-packages \
398  --default-builds=MPI_DEBUG
399 
400  NOTE: This will do only an MPI DEBUG build and will only build and run the
401  tests for the packages that have directly been changed and not any forward
402  packages. Replace "MPI_DEBUG" with whatever your most important default
403  build is.
404 
405 (*) Test only a specific set of packages and no others:
406 
407  ../checkin-test.py \
408  --enable-packages=<P0>,<P1>,<P2> --no-enable-fwd-packages \
409  --do-all
410 
411  NOTE: This will override all logic in the script about which packages will
412  be enabled based on file changes and only the given packages will be
413  enabled. When there are tens of thousands of changed files and hundreds of
414  defined packages, this auto-detection algorithm can be very expensive!
415 
416  NOTE: You might also want to pass in --enable-all-packages=off in case the
417  script wants to enable all the packages (see the output in the
418  checkin-test.py log file for details) and you think it is not necessary to
419  do so.
420 
421  NOTE: Using these options is greatly preferred to not running this script at
422  all and should not be any more expensive than the testing you would already
423  do manually before a push.
424 
425 (*) Test changes locally without pulling updates:
426 
427  ../checkin-test.py --local-do-all
428 
429  NOTE: This will just configure, build, test, and send an email notification
430  without updating or changing the status of the local git repo in any way and
431  without any communication with the global repo. Hence, you can have
432  uncommitted changes and still run configure, build, test without having to
433  commit or having to stash changes.
434 
435  NOTE: This will determine what packages to enable and test based on changes
436  w.r.t. to the tracking branch. If not on a tracking branch, or in a
437  detached head state, see below.
438 
439  NOTE: This is typically not a sufficient level of testing in order to push
440  the changes to a shared branch because you have not fully integrated your
441  changes yet with other developers. However, this would be a sufficient
442  level of testing in order to do a commit on the local machine and then pull
443  to a remote machine for further testing and a push (see below).
444 
445 (*) Local test of repo version on a detached head or with no tracking branch:
446 
447  ../checkin-test.py --enable-all-packages=[on|off] \
448  --enable-packages=<P0>,... --local-do-all
449 
450  By specifying what packages are enabled and not doing a pull or push, the
451  script allows the repo(s) to be in a detached head state or on a branch that
452  does not have a tracking branch. This allows the checkin-test.py script to
453  be used, for example, to test versions using 'git bisect'.
454 
455 (*) Adding extra build/test cases:
456 
457  Often you will be working on Secondary Tested (ST) Code or Experimental (EX)
458  Code and want to include the testing of this in your pre-push testing
459  process along with the standard --default-builds build/test cases which can
460  only include Primary Tested (PT) Code. In this case you can run with:
461 
462  ../checkin-test.py --extra-builds=<BUILD1>,<BUILD2>,... [other options]
463 
464  For example, if you have a build that enables the TPL CUDA you would do:
465 
466  echo "
467  -DTPL_ENABLE_MPI:BOOL=ON
468  -DTPL_ENABLE_CUDA:BOOL=ON
469  " > MPI_DEBUG_CUDA.config
470 
471  and then run with:
472 
473  ../checkin-test.py --extra-builds=MPI_DEBUG_CUDA --do-all
474 
475  This will do the standard --default-builds (e.g. MPI_DEBUG and
476  SERIAL_RELEASE) build/test cases along with your non-standard MPI_DEBUG_CUDA
477  build/test case.
478 
479  NOTE: You can disable the default build/test cases with --default-builds="".
480  However, please only do this when you are not going to push because you need
481  at least one default build/test case (the most important default PT case,
482  e.g. MPI_DEBUG) to do a safe push.
483 
484 (*) Including extra repos and extra packages:
485 
486  You can also use the checkin-test.py script to continuously integrate
487  multiple git repos containing add-on packages. To do so, just run:
488 
489  ../checkin-test.py --extra-repos=<REPO1>,<REPO2>,... [options]
490 
491  NOTE: You have to create local commits in all of the extra repos where there
492  are changes or the script will abort.
493 
494  NOTE: Extra repos can be specified with more flexibility using the
495  --extra-repos-file and --extra-repos-type arguments (also see
496  --ignore-missing-extra-repos).
497 
498  NOTE: Each of the last local commits in each of the changed repos will get
499  amended with the appended summary of what was enabled in the build/test (if
500  --append-test-results is set).
501 
502 (*) Avoid changing any of the local commit SHA1s:
503 
504  If you are pushing commits from a shared branch, it is critical that you do
505  not change any of the SHA1s of the commits. Changing the SHA1s for any of
506  the commits will mess up various multi-repo, multi-branch workflows. To
507  avoid changing any of the SHA1s of the local commits, one must run with:
508 
509  ../checkin-test.py --no-rebase --no-append-test-results [options]
510 
511 (*) Performing a remote test/push:
512 
513  If you develop on a slow machine like your laptop, doing an appropriate
514  level of testing can take a long time. In this case, you can pull the
515  changes to another faster remote workstation and do a more complete set of
516  tests and push from there. If you are knowledgeable with git, this will be
517  easy and natural to do, without any help from this script. However, this
518  script can still help and automate the steps and can do so in one command
519  invocation on the part of the developer.
520 
521  On your slow local development machine 'mymachine', do the limited testing
522  with:
523 
524  ../checkin-test.py --do-all --no-enable-fwd-packages
525 
526  On your fast remote test machine, do a full test and push with:
527 
528  ../checkin-test.py \
529  --extra-pull-from=<remote-repo>:master \
530  --do-all --push
531 
532  where <remote-name> is a git remote repo name pointing to
533  mymachine:/some/dir/to/your/src (see 'git help remote').
534 
535  NOTE: You can of course adjust the packages and/or build/test cases that get
536  enabled on the different machines.
537 
538  NOTE: Once you invoke the checkin-test.py script on the remote test machine
539  and it has pulled the commits from mymachine, then you can start changing
540  files again on your local development machine and just check your email to
541  see what happens on the remote test machine.
542 
543  NOTE: If something goes wrong on the remote test machine, you can either
544  work on fixing the problem there or you can fix the problem on your local
545  development machine and then do the process over again.
546 
547  NOTE: If you alter the commits on the remote machine (such as squashing
548  commits), you will have trouble merging back on our local machine.
549  Therefore, if you have to to fix problems, make new commits and don't alter
550  the ones you pulled from your local machine (but rebasing them should be
551  okay as long as the local commits on mymachine are not pushed to other
552  repos).
553 
554  NOTE: Git will resolve the duplicated commits when you pull the commits
555  pushed from the remote machine. Git knows that the commits are the same and
556  will do the right thing when rebasing (or just merging).
557 
558  NOTE: This would also work for multiple repos if the remote name
559  '<remote-repo>' pointed to the right remote repo in all the local repos.

604 HANDLING OF PT, ST, AND EX CODE IN BUILT-IN AND EXTRA BUILDS:
605 -------------------------------------------------------------
606 
607 This script will only process PT (Primary Tested) packages in the
608 --default-builds (e.g. MPI_DEBUG and SERIAL_RELEASE) builds. This is to avoid
609 problems of side-effects of turning on ST packages that would impact PT
610 packages (e.g. an ST package getting enabled that enables an ST TPL which
611 turns on support for that TPL in a PT package producing different code which
612 might work but the pure PT build without the extra TPL may actually be broken
613 and not know it). Therefore, any non-PT packages that are enabled (either
614 implicitly through changed files or explicitly by listing in --enable-packages)
615 will be turned off in the --default-builds builds. If none of the enabled
616 packages are PT, then they will all be disabled and the --default-builds
617 builds will be skipped.
618 
619 In order to better support the development of ST and EX packages, this script
620 allows you to define some extra builds that will be invoked and used to
621 determine overall pass/fail before a potential push. The option
622 --st-extra-builds is used to specify extra builds that will test ST packages
623 (and also PT packages if any are enabled). If only PT packages are enabled
624 then the builds specified in --st-extra-builds will still be run. The
625 reasoning is that PT packages may contain extra ST features and therefore if
626 the goal is to test these ST builds it is desirable to also run these builds
627 because they also my impact downstream ST packages.
628 
629 Finally, the option --extra-builds will test all enabled packages, including
630 EX packages, regardless of their test group. Therefore, when using
631 --extra-builds, be careful that you watch what packages are enabled. If you
632 change an EX package, it will be enabled in --extra-builds builds.
633 
634 A few use cases might help better demonstrate the behavior. Consider
635 the following input arguments specifying extra builds
636 
637  --st-extra-builds=MPI_DEBUG_ST --extra-builds=INTEL_DEBUG
638 
639 with the packages Teuchos, Phalanx, and Meros where Teuchos is PT, Phalanx is
640 ST, and Meros is EX.
641 
642 Here is what packages would be enabled in each of the builds:
643

645  --st-extra-builds=MPI_DEBUG_ST \
646  --extra-builds=INTEL_DEBUG
647 
648 and which packages would be excluded:
649 
650 A) --enable-packages=Teuchos:
651  MPI_DEBUG: [Teuchos]
652  SERIAL_RELEASE: [Teuchos]
653  MPI_DEBUG_ST: [Teuchos]
654  INTEL_DEBUG: [Teuchos] Always enabled!
655 
656 B) --enable-packages=Phalanx:
657  MPI_DEBUG: [] Skipped, no PT packages!
658  SERIAL_RELEASE: [] Skipped, no PT packages!
659  MPI_DEBUG_ST: [Phalanx]
660  INTEL_DEBUG: [Phalanx]
661 
662 C) --enable-packages=Meros:
663  MPI_DEBUG: [] Skipped, no PT packages!
664  SERIAL_RELEASE: [] Skipped, no PT packages!
665  MPI_DEBUG_ST: [] Skipped, no PT or ST packages!
666  INTEL_DEBUG: [Meros]
667 
668 D) --enable-packages=Teuchos,Phalanx:
669  MPI_DEBUG: [Teuchos]
670  SERIAL_RELEASE: [Teuchos]
671  MPI_DEBUG_ST: [Teuchos,Phalanx]
672  INTEL_DEBUG: [Teuchos,Phalanx]

676  SERIAL_RELEASE: [Teuchos]
677  MPI_DEBUG_ST: [Teuchos,Phalanx]
678  INTEL_DEBUG: [Teuchos,Phalanx,Meros]
679 
680 The --extra-builds=INTEL_DEBUG build is always performed with all of the
681 enabled packages. This logic given above must be understood in order to
